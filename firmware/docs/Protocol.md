# 1. Purpose

This document defines the USB communication protocol between the Host System and the NAS Controller Firmware running on the ESP32.

The protocol standardizes communication independently of the operating system. Any Host Software implementing this specification SHALL be capable of communicating with any firmware implementing the same protocol version.

The protocol SHALL remain stable throughout all Firmware Version 1.x releases. Breaking protocol changes require a new Protocol Major Version.

---

# 2. Scope

This protocol specifies communication for:

- Hardware monitoring
- Relay control
- Fan control
- Temperature monitoring
- Drive activity
- LED status
- Firmware information
- Configuration
- Events
- Error reporting

The protocol intentionally excludes:

- File transfer
- Firmware updates
- Debug console
- Operating system APIs
- Network communication

These functions SHALL be handled independently of the protocol.

---

# 3. Definitions

- Host: The computer executing the Host Service.
- ESP32: The embedded controller responsible for hardware management.
- Firmware: The software executing on the ESP32.
- Host Service: Software executing on the Host Computer implementing this protocol.
- Protocol: The set of communication rules defined by this document.
- Message: A single line of communication transmitted between Host and ESP32.
- Session: Communication beginning after successful connection and ending upon disconnection.

---

# 4. Terminology

The following terminology is used throughout this specification.

- MUST: Mandatory requirement.
- SHALL: Equivalent to MUST.
- SHOULD: Strong recommendation.
- MAY: Optional behavior.
- HOST: The controlling computer.
- DEVICE: The ESP32 NAS Controller.
- COMMAND: Instruction transmitted by the Host.
- RESPONSE: Reply transmitted by the Device.
- EVENT: Asynchronous notification generated by the Device.
- STATUS: Current operational information.
- OBJECT: Hardware resource or logical entity.

Examples of objects include relay, fan, temperature sensor, LED, and drive.

# 5. Design Principles

The protocol has been designed according to the following principles.

## 5.1 Human Readable

Every message SHALL be readable using a standard serial terminal.

Example:

```text
STATUS SYSTEM
GET TEMP
SET RELAY ID=1 STATE=ON
```

## 5.2 Cross Platform

The protocol SHALL NOT depend on any operating system. Identical firmware SHALL operate with Windows, TrueNAS, OpenMediaVault, Debian, and future operating systems without firmware modification.

## 5.3 Versioned

Every firmware SHALL report the firmware version, protocol version, and hardware version during initialization.

## 5.4 Backward Compatibility

Firmware implementing Protocol Version 1.x SHOULD remain compatible with Host Software implementing Protocol Version 1.x. Breaking changes require Protocol Version 2.0.

## 5.5 Extensible

Future commands SHALL NOT invalidate existing commands. Reserved command ranges SHALL be maintained.

## 5.6 Deterministic

The same command SHALL always produce the same response under identical conditions.

## 5.7 Hardware Ownership

The protocol defines a strict architectural boundary:

- The Host reports facts.
- The ESP32 decides hardware actions.

Host responsibilities include:

- Drive monitoring
- SMART monitoring
- Operating system integration
- Backup state
- Host events

ESP32 responsibilities include:

- Relay control
- Fan control
- Temperature monitoring
- LED behaviour
- Hardware protection
- Failsafe operation
- Watchdog recovery

The ESP32 SHALL NOT perform operating-system-specific functions. The Host SHALL NOT directly manipulate hardware GPIO.

# 6. Transport Layer

## 6.1 Physical Interface

USB 2.0 Full Speed over USB CDC (Virtual COM Port). No vendor-specific drivers are required.

## 6.2 Communication Direction

Bidirectional: Host ↔ ESP32.

## 6.3 Encoding

ASCII, UTF-8 compatible, with printable characters only.

## 6.4 Line Termination

Each message SHALL terminate using LF. The firmware SHOULD accept LF or CRLF for compatibility.

## 6.5 Message Size

Maximum message length is 256 characters. Messages exceeding this limit SHALL be rejected.

## 6.6 Character Set

Allowed characters are A-Z, 0-9, space, =, _, and -. Binary characters, control characters, tabs, and Unicode symbols are forbidden.

## 6.7 Message Structure

Every message consists of COMMAND, OBJECT, and PARAMETERS.

Examples:

```text
PING
GET STATUS
GET VERSION
SET RELAY ID=1 STATE=ON
SET FAN MODE=AUTO
GET TEMP SENSOR=2
EVENT DRIVE ID=3 STATE=READING
```

## 6.8 Case Sensitivity

Commands SHALL be uppercase. Parameter names SHALL be uppercase. Parameter values SHALL be uppercase unless otherwise specified.

Examples:

```text
Correct: SET RELAY ID=1 STATE=ON
Incorrect: set relay
Incorrect: Set Relay
```

## 6.9 Reserved Characters

The following characters are reserved: =, space, :, -, and _. These characters SHALL only be used according to the syntax rules defined by this specification.

# 7. Connection Lifecycle

## 7.1 Overview

Every communication session between the Host and the ESP32 SHALL follow the lifecycle defined in this section. The purpose of the lifecycle is to ensure that both devices always know the current communication state. The protocol SHALL support automatic recovery from unexpected USB disconnections.

## 7.2 Connection States

The Host SHALL transition through the following states:

```text
DISCONNECTED
  -> PORT OPEN
  -> HANDSHAKE
  -> VERSION CHECK
  -> READY
  -> NORMAL OPERATION
  -> DISCONNECTED
```

The ESP32 SHALL transition through the following states:

```text
BOOT
  -> USB WAIT
  -> HANDSHAKE
  -> READY
  -> NORMAL OPERATION
  -> USB LOST
  -> USB WAIT
```

## 7.3 Boot Sequence

When power is applied to the ESP32, the firmware SHALL initialize hardware, drivers, services, and the protocol manager, and then wait for the USB Host. The ESP32 SHALL NOT assume that a Host is connected.

## 7.4 Host Connection

When the Host opens the USB serial port, the Host SHALL transmit HELLO and then wait for a response.

Example:

```text
HELLO
```

## 7.5 Device Response

Upon receiving HELLO, the ESP32 SHALL respond with a structured handshake response.

Example:

```text
HELLO
PROTOCOL=1.0
FIRMWARE=0.0.2
HARDWARE=1.0
READY
END
```

The READY message indicates that the firmware has completed initialization.

## 7.6 Version Verification

The Host SHALL compare the protocol version reported by the ESP32 with the protocol version supported by the Host. If compatible, communication continues. If incompatible, the Host SHALL disconnect.

Example:

```text
ERROR PROTOCOL VERSION
```

## 7.7 Communication Ready

Once READY has been received, both Host and ESP32 SHALL enter NORMAL OPERATION. No commands SHALL be transmitted before READY.

## 7.8 Unexpected Disconnect

If USB communication is interrupted, both Host and ESP32 SHALL immediately enter the DISCONNECTED state. Pending commands SHALL be discarded. No automatic command replay SHALL occur.

## 7.9 Reconnection

After USB reconnect, the complete handshake SHALL be repeated. No previous session state SHALL be assumed.

# 8. Message Format

## 8.1 General Structure

Every protocol message SHALL consist of COMMAND, OBJECT, and PARAMETERS.

Example:

```text
SET RELAY ID=1 STATE=ON
GET TEMP SENSOR=2
EVENT DRIVE ID=3 STATE=READING
```

## 8.2 Command

The first word SHALL always identify the command.

Examples:

- GET
- SET
- EVENT
- ACK
- CONFIG
- PING

## 8.3 Object

The second word identifies the hardware or logical object.

Examples:

- SYSTEM
- RELAY
- FAN
- TEMP
- LED
- DRIVE
- VERSION
- CONFIG
- STATUS

## 8.4 Parameters

Parameters SHALL use KEY=VALUE format.

Examples:

```text
ID=1
STATE=ON
MODE=AUTO
SENSOR=2
VALUE=36
RPM=1250
```

## 8.5 Parameter Order

Parameter order SHOULD NOT matter.

Example:

```text
SET RELAY STATE=ON ID=1
```

This is equivalent to:

```text
SET RELAY ID=1 STATE=ON
```

## 8.6 Whitespace

One or more spaces MAY separate fields. Leading spaces SHALL be ignored. Trailing spaces SHALL be ignored.

## 8.7 Empty Messages

Empty messages SHALL be ignored. No response SHALL be generated.

## 8.8 Unknown Parameters

Unknown parameters SHALL be ignored unless they affect command interpretation.

Example:

```text
SET FAN MODE=AUTO TEST=123
```

The TEST parameter is ignored.

## 8.9 Invalid Syntax

Malformed commands SHALL generate:

```text
ERROR INVALID COMMAND
```

## 8.10 Maximum Parameters

There is no fixed parameter count. Each command defines its required parameters.

# 9. Protocol Parameter Naming Convention

This section defines the standardized parameter names used throughout Protocol Version 1.x. These names SHALL be used consistently in all command examples and responses unless a command defines a specific object-specific field.

## 9.1 Reserved Parameter Names

| Parameter | Purpose | Allowed values | Example | Meaning |
| --- | --- | --- | --- | --- |
| ID | Unique object identifier. | Numeric identifier or implementation-defined object ID. | ID=1 | Identifies a specific relay, fan, sensor, or drive. |
| STATE | Current operational state. | Device-specific states such as ON, OFF, READY, ACTIVE, BUSY, WARNING, CRITICAL, ERROR, READING, WRITING, STANDBY, or UNKNOWN. | STATE=ON | Indicates the current state of the target object. |
| MODE | Operating mode. | AUTO, MANUAL, SOLID, BLINK, PULSE, OFF, FAILSAFE, NORMAL, or SAFE. | MODE=AUTO | Selects or reports the operating mode. |
| VALUE | Measured or configured numeric value. | Numeric values, typically decimal. | VALUE=34.8 | Reports a measurement or configured numeric setting. |
| UNIT | Measurement unit. | C, F, %, or other implementation-defined units. | UNIT=C | Indicates the unit associated with VALUE. |
| ENABLE | Boolean enable flag. | ON, OFF, TRUE, FALSE, 1, 0. | ENABLE=ON | Enables or disables a feature or function. |
| TIMEOUT | Timeout value in milliseconds. | Integer value in milliseconds. | TIMEOUT=1000 | Defines a timeout interval. |
| SENSOR | Sensor identifier. | Numeric sensor index. | SENSOR=2 | Identifies a specific temperature sensor. |
| COLOR | Visual color value. | OFF, WHITE, RED, GREEN, BLUE, YELLOW, CYAN, MAGENTA. | COLOR=GREEN | Selects or reports LED color. |
| PWM | Pulse-width modulation value. | Integer from 0 to 100. | PWM=65 | Reports or configures fan PWM. |
| RPM | Rotational speed. | Integer revolutions per minute. | RPM=1320 | Reports fan rotational speed. |
| PROTOCOL | Protocol version identifier. | Protocol major.minor format. | PROTOCOL=1.0 | Reports the protocol version. |
| FIRMWARE | Firmware version identifier. | Semantic version string. | FIRMWARE=0.0.2 | Reports firmware version information. |
| HARDWARE | Hardware revision identifier. | Implementation-defined hardware revision. | HARDWARE=1.0 | Reports hardware revision. |
| BUILD | Build identifier. | String or numeric build identifier. | BUILD=20260717 | Identifies the firmware build. |

## 9.2 Naming Rules

- Parameter names SHALL be uppercase.
- Parameter values SHALL be uppercase unless otherwise specified.
- Parameter names SHALL remain stable across Protocol Version 1.x.
- New parameters SHALL use the reserved names above whenever applicable.

# 10. Host and Device Responsibilities

This chapter freezes the architectural rule for Protocol Version 1.x:

> The Host reports facts. The ESP32 decides hardware actions.

This rule SHALL remain unchanged throughout Protocol Version 1.x.

## 10.1 Host Responsibilities

The Host SHALL be responsible for reporting system facts and events, including:

- Drive monitoring
- SMART monitoring
- Operating system integration
- Backup state
- Host events

The Host may report such information to the ESP32, but it SHALL NOT directly control hardware outputs.

## 10.2 ESP32 Responsibilities

The ESP32 SHALL be responsible for deciding and executing hardware actions, including:

- Relay control
- Fan control
- Temperature monitoring
- LED behaviour
- Hardware protection
- Failsafe operation
- Watchdog recovery

The ESP32 SHALL make hardware safety decisions independently of the Host.

## 10.3 Examples

Correct:

```text
EVENT DRIVE ID=3 STATE=WRITING
```

Incorrect:

```text
SET LED COLOR=GREEN
```

The first example reports a host-observed drive state. The second example attempts to direct firmware behaviour through a command that belongs to the firmware-owned decision domain. LED behaviour, fan behaviour, and hardware safety decisions belong exclusively to the firmware.

---

# 11. Host State Machine

## 11.1 Purpose

The Host controls protocol communication. The Host SHALL always initiate commands. The ESP32 SHALL NEVER issue commands spontaneously except EVENT messages.

## 11.2 Host States

- DISCONNECTED
- USB CLOSED
- PORT OPEN
- HANDSHAKE
- VERSION CHECK
- READY
- ACTIVE
- ERROR
- RECONNECT

## 11.3 Host Responsibilities

The Host SHALL:

- Monitor drive activity
- Monitor SMART status
- Monitor backup operations
- Request relay state changes
- Request LED state changes
- Request fan operating mode changes
- Monitor firmware heartbeat
- Reconnect automatically when required
- Log communication

## 11.4 Command Queue

The Host SHOULD transmit commands sequentially. A command SHOULD complete before another command is issued. Future protocol versions MAY support command pipelining.

## 11.5 Timeout

If no response is received within the configured timeout, the Host SHALL retry. After three failures, the Host SHALL disconnect and restart the handshake.

---

# 12. ESP32 State Machine

## 12.1 Purpose

The ESP32 manages hardware. It SHALL execute only valid commands received from the Host.

## 12.2 Device States

- BOOT
- WAIT HOST
- READY
- ACTIVE
- BUSY
- ERROR
- SAFE MODE

## 12.3 Safe Mode

The firmware SHALL enter Safe Mode when an internal hardware fault, fatal initialization error, watchdog recovery event, critical temperature, or unexpected hardware failure occurs. In Safe Mode, relays SHALL remain OFF unless safety requires otherwise. Fans SHALL operate using the configured failsafe policy. LEDs SHALL indicate an ERROR state. Only diagnostic commands SHALL be accepted.

## 12.4 Event Generation

The ESP32 MAY generate EVENT messages asynchronously.

Examples:

```text
EVENT TEMP SENSOR=1 VALUE=42
EVENT FAN ID=1 RPM=1200
EVENT RELAY ID=2 STATE=OFF
EVENT ERROR CODE=102
```

## 12.5 Internal Independence

The ESP32 SHALL continue temperature monitoring, fan control, safety monitoring, and watchdog supervision even if USB communication is lost. Loss of Host communication SHALL NOT stop hardware safety functions.

# 13. Command Set

## 13.1 Overview

The Host communicates with the ESP32 using commands. Every command SHALL receive exactly one response. Commands SHALL be processed sequentially. Unless otherwise specified, command execution SHALL be atomic.

## 13.2 Command Categories

- Session Commands
- System Commands
- Version Commands
- Relay Commands
- Fan Commands
- Temperature Commands
- LED Commands
- Drive Commands
- Configuration Commands
- Diagnostic Commands

## 13.3 Session Commands

### HELLO

Purpose: Initiate protocol handshake.

Request:

```text
HELLO
```

Response:

```text
HELLO
PROTOCOL=1.0
FIRMWARE=0.0.2
HARDWARE=1.0
READY
END
```

### ACK

Purpose: Reserved for future acknowledgement semantics.

This command is reserved for Protocol Version 1.x and SHALL NOT be used by Protocol Version 1.0 implementations.

### PING

Purpose: Verify communication.

Request:

```text
PING
```

Response:

```text
PONG
```

### RESET

Purpose: Restart ESP32 firmware.

Request:

```text
RESET
```

Response:

```text
OK RESET
```

The firmware SHALL reboot after transmitting the response.

## 13.4 System Commands

### GET SYSTEM

Purpose: Retrieve overall controller status.

Request:

```text
GET SYSTEM
```

Response:

```text
SYSTEM
STATE=ACTIVE
UPTIME=8453
VALUE=36
FANS=2
RELAYS=2
DRIVES=10
END
```

### GET STATUS

Purpose: Retrieve complete system status.

Response SHALL include firmware status, relay status, fan status, temperature, and protocol version.

## 13.5 Version Commands

### GET VERSION

Purpose: Retrieve firmware version information.

Request:

```text
GET VERSION
```

Response:

```text
VERSION
FIRMWARE=0.0.2
PROTOCOL=1.0
HARDWARE=1.0
BUILD=20260717
END
```

## 13.6 Relay Commands

### GET RELAY

Purpose: Retrieve relay state.

Request:

```text
GET RELAY ID=1
```

Response:

```text
RELAY
ID=1
STATE=ON
END
```

### SET RELAY

Purpose: Control relay output.

Request:

```text
SET RELAY ID=1 STATE=ON
```

Response:

```text
OK
```

Invalid relay IDs SHALL return:

```text
ERROR INVALID ID
```

## 13.7 Fan Commands

### GET FAN

Purpose: Retrieve fan information.

Request:

```text
GET FAN ID=1
```

Response:

```text
FAN
ID=1
MODE=AUTO
PWM=65
RPM=1280
END
```

### SET FAN

Purpose: Configure fan behaviour.

Examples:

```text
SET FAN ID=1 MODE=AUTO
SET FAN ID=1 MODE=MANUAL PWM=75
```

## 13.8 Temperature Commands

### GET TEMP

Purpose: Read a temperature sensor.

Request:

```text
GET TEMP SENSOR=1
```

Response:

```text
TEMP
SENSOR=1
VALUE=34.8
UNIT=C
END
```

### GET TEMPS

Purpose: Read all sensors.

Request:

```text
GET TEMPS
```

Response:

```text

SENSOR=3 VALUE=33.8
END
```

## 13.9 LED Commands

### GET LED

Request:

```text
GET LED ID=1
```

```text

COLOR=GREEN
MODE=SOLID
END
```

### SET LED

Request:

```text
SET LED ID=1 COLOR=GREEN MODE=BLINK
```

Response:

```text

```

## 13.10 Drive Commands

### GET DRIVE

Purpose: Retrieve drive state.

Request:

```text
GET DRIVE ID=3
```

Response:

```text
DRIVE
ID=3
STATE=READING
END
```

## 13.11 Configuration Commands

### GET CONFIG

Purpose: Retrieve firmware configuration.

Request:

```text
GET CONFIG
```

Response:

```text
CONFIG
MODE=AUTO
VALUE=45

TIMEOUT=1000
END
```

### SET CONFIG

Purpose: Modify configuration.

Examples:


## 13.12 Diagnostic Commands

- GET LOG: Reserved for future implementation.
- GET ERROR: Retrieve latest error.
- CLEAR ERROR: Clear recoverable errors.

# 14. Response Set

## 14.1 General

Every command SHALL produce one response. Responses SHALL begin with OK, ERROR, or an object name. Structured data responses SHALL be encoded as multi-line blocks terminated by END.

## 14.2 OK Response

Simple acknowledgements SHALL be a single line.

Examples:

```text
OK
OK RELAY
OK CONFIG SAVED
```

## 14.3 ERROR Response

Format:

```text
ERROR <Description>
```

Examples:

```text
ERROR INVALID COMMAND
ERROR INVALID PARAMETER
ERROR DEVICE BUSY
```

## 14.4 Object Response

Object responses SHALL use the structured format shown below.

Example:

```text
FAN
ID=1
RPM=1320
PWM=70
MODE=AUTO
END
```

## 14.5 Multi-Line Responses

Multi-line responses SHALL terminate with END.

Example:

```text
TEMP
SENSOR=1 VALUE=34
SENSOR=2 VALUE=35
SENSOR=3 VALUE=36
END
```

# 17. Drive States

## 17.1 Overview

The Host Software is responsible for determining drive activity. The ESP32 SHALL NOT independently determine drive activity. The Host SHALL periodically report drive status to the ESP32.

## 17.2 Drive Identifier

Each drive SHALL have a unique identifier.

Example:

```text
ID=1
ID=2
ID=3
...
ID=10
```

Drive IDs SHALL remain stable for the duration of a session.

## 17.3 Supported Drive States

- OFF
- POWERING_ON
- INITIALIZING
- READY
- READING
- WRITING
- BUSY
- STANDBY
- SLEEP
- ERROR
- UNKNOWN

## 17.4 Reporting Drive State

```text
EVENT DRIVE
ID=3
STATE=READING
```

```text
EVENT DRIVE
ID=4
STATE=WRITING
```

```text
EVENT DRIVE
ID=8
STATE=STANDBY
```

## 17.5 Drive Attributes

Future protocol versions MAY include TEMPERATURE, SMART, MODEL, SERIAL, CAPACITY, and HEALTH. These parameters SHALL NOT modify existing protocol behavior.

# 18. Relay Control

## 18.1 Purpose

Relay outputs control external hardware. Typical applications include backup drive power and peripheral power.

## 18.2 Relay Identification

Every relay SHALL have an ID and STATE.

Example:

```text
ID=1
STATE=ON
```

## 18.3 Relay States

- OFF
- ON
- UNKNOWN

## 18.4 Commands

Turn relay ON:

```text
SET RELAY ID=1 STATE=ON
```

Turn relay OFF:

```text
SET RELAY ID=1 STATE=OFF
```

Read relay:

```text
GET RELAY ID=1
```

## 18.5 Safety Rules

The firmware SHALL validate the relay ID, requested state, and hardware availability. Invalid requests SHALL be rejected.

## 18.6 Future Expansion

Reserved parameters such as DELAY, TIMEOUT, and AUTO SHALL be ignored by Protocol Version 1.0 firmware.

# 19. Fan Control

## 19.1 Overview

The ESP32 owns fan control. The Host requests an operating mode. The ESP32 determines PWM output.

## 19.2 Fan Modes

- AUTO
- MANUAL
- OFF
- FAILSAFE

## 19.3 Manual Mode

Example:

```text
SET FAN ID=1 MODE=MANUAL PWM=65
```

PWM represents the fan duty cycle percentage. The allowed range is 0 to 100.

## 19.4 Automatic Mode

Example:

```text
SET FAN ID=1 MODE=AUTO
```

The ESP32 SHALL determine PWM.

## 19.5 Fan Status

```text
FAN
ID=1
MODE=AUTO
PWM=54
RPM=1320
STATE=RUNNING
END
```

## 19.6 Fan States

- OFF
- RUNNING
- STOPPED
- STALL
- ERROR
- UNKNOWN

## 19.7 Fan Failure

If RPM falls below the configured threshold, the ESP32 SHALL generate:

```text
EVENT FAN
ID=1
STATE=ERROR
```

# 20. Temperature Reporting

## 20.1 Overview

Temperature sensors are managed by the ESP32. The Host MAY request current temperatures. The ESP32 MAY generate temperature events.

## 20.2 Temperature Units

Protocol Version 1.0 SHALL use degrees Celsius, with UNIT=C. Future versions MAY support Fahrenheit.

## 20.3 Sensor Identification

Sensors SHALL use the SENSOR parameter.

Examples:

```text
SENSOR=1
SENSOR=2
SENSOR=3
```

## 20.4 Reading Temperature

Request:

```text
GET TEMP SENSOR=2
```

Response:

```text
TEMP
SENSOR=2
VALUE=34.6
UNIT=C
END
```

## 20.5 Reading All Sensors

Request:

```text
GET TEMPS
```

Response:

```text
TEMP
SENSOR=1 VALUE=32.4
SENSOR=2 VALUE=35.8
SENSOR=3 VALUE=34.1
END
```

## 20.6 Temperature States

- NORMAL
- WARNING
- CRITICAL
- UNKNOWN

## 20.7 Warning Threshold

If the configured warning temperature is exceeded, the ESP32 SHALL generate:

```text
EVENT TEMP
SENSOR=2
STATE=WARNING
VALUE=46
```

## 20.8 Critical Temperature

If the configured critical temperature is exceeded, the ESP32 SHALL increase fan speed, generate an event, and enter failsafe if required.

Example:

```text
EVENT TEMP
SENSOR=2
STATE=CRITICAL
VALUE=58
```

Protocol Version 1.0 defines the following colors:

# 23. Heartbeat

## 23.1 Purpose

Heartbeat messages verify that communication between Host and ESP32 remains active.

## 23.2 Interval

The default interval is 1000 ms and MAY be configured.

## 23.3 Host Heartbeat

The Host transmits PING. The ESP32 responds with PONG.

## 23.4 Heartbeat Failure

If three consecutive heartbeat intervals are missed, the ESP32 SHALL generate a communication timeout, enter the Host Disconnected state, and continue hardware safety functions. The Host SHALL close communication, reconnect, and repeat the handshake.

---

# 24. Timing Requirements

## 24.1 Command Response

Normal commands SHALL be responded to within 100 ms.

## 24.2 Long Operations

Commands requiring longer execution SHALL immediately return OK BUSY. The final completion status SHALL be transmitted using EVENT.

## 24.3 USB Recovery

USB reconnection SHALL complete using the normal handshake. No session recovery mechanism exists.

## 24.4 Event Latency

Hardware-generated events SHOULD be transmitted within 100 ms.

## 24.5 Temperature Update

The default temperature update interval is 1000 ms and is configurable.

## 24.6 Fan Update

The default fan update interval is 500 ms and is configurable.

---

# 25. Security Considerations

## 25.1 Overview

Protocol Version 1.0 is intended for communication between trusted devices connected through USB. Authentication and encryption are not implemented.

## 25.2 Invalid Commands

Malformed commands SHALL NOT cause firmware crashes. Unknown commands SHALL return:

```text
ERROR UNKNOWN COMMAND
```

## 25.3 Invalid Parameters

Out-of-range values SHALL be rejected.

Example:

```text
SET FAN VALUE=150
```

Response:

```text
ERROR INVALID VALUE
```

## 25.4 Buffer Protection

The firmware SHALL validate message length, parameter count, numeric conversions, and string length before processing.

## 25.5 Watchdog

The firmware SHALL use the ESP32 watchdog timer. Unexpected firmware lockups SHALL automatically restart the controller.

# 26. Future Expansion

Protocol Version 1.x reserves command ranges for future functionality. Reserved modules include SMART, UPS, OLED Display, LCD Display, Ethernet, Wi-Fi, MQTT, Home Assistant, OTA Update, RTC, Power Meter, Drive Identification, RGB LED, Statistics, Logging, and Diagnostics. Reserved keywords SHALL NOT be used by Protocol Version 1.0.

## E. Compatibility Policy

Protocol Version 1.x firmware SHALL remain compatible with Protocol Version 1.x Host Software. Breaking protocol changes require Protocol Version 2.0.

## F. Formal Message Grammar

The following EBNF-style grammar describes the protocol structure.

```ebnf
<message> ::= <command> [<object>] [<parameter-list>]

<command> ::= "GET"
            | "SET"
            | "EVENT"
            | "ACK"
            | "HELLO"
            | "PING"
            | "RESET"
            | "CONFIG"

<object> ::= "SYSTEM"
           | "STATUS"
           | "VERSION"
           | "RELAY"
           | "FAN"
           | "TEMP"
           | "LED"
           | "DRIVE"
           | "CONFIG"

<parameter-list> ::= <parameter>*

<parameter> ::= <key> "=" <value>

<key> ::= "ID"
        | "STATE"
        | "MODE"
        | "VALUE"
        | "UNIT"
        | "ENABLE"
        | "TIMEOUT"
        | "SENSOR"
        | "COLOR"
        | "PWM"
        | "RPM"
        | "PROTOCOL"
        | "FIRMWARE"
        | "HARDWARE"
        | "BUILD"
        | <other-key>

<value> ::= <token>

<token> ::= <character> { <character> }
<character> ::= "A" | "B" | ... | "Z" | "0" | "1" | ... | "9" | "-" | "_"
```
